## 1. 产品概述

### 1.1 产品定位

**InputLock for macOS 26.2** 是一款面向开源社区的轻量级输入法锁定工具。产品采用"监听-纠正"技术模式，解决 macOS 用户在特定场景下意外切换输入法的痛点问题。

### 1.2 目标用户

| 用户类型 | 特征 | 核心诉求 |
|---------|------|---------|
| 程序员 | 大量英文代码编写，偶尔中文注释 | 编程时保持英文输入法，避免中文干扰 |
| 内容创作者 | 中英文混排写作 | 写作时保持特定输入法，提高效率 |
| 多语言办公者 | 频繁切换工作场景 | 不同场景下固定输入法状态，减少误操作 |
| 游戏玩家 | 游戏中需要固定输入法 | 防止游戏中输入法切换影响操作 |

### 1.3 核心价值

- **零学习成本**：菜单栏图标一键锁定/解锁，符合 macOS 原生操作习惯
- **无侵入性**：无需系统权限，使用公开 API，安全可靠
- **即时响应**：50 毫秒纠正延迟，用户几乎无感知

### 1.4 产品目标

| 月活跃用户 | 500+ |

## 2. 功能需求

### 2.1 核心功能矩阵

| 功能模块 | 优先级 | 说明 |
|---------|--------|------|
| 输入法枚举 | P0 | 自动检测并显示系统中所有可用输入法 |
| 一键锁定 | P0 | 点击输入法项即可切换并锁定 |
| 锁定状态管理 | P0 | 全局锁定/解锁切换 |
| 菜单栏常驻 | P0 | 状态栏图标显示当前锁定状态（🔓/🔒） |
| 多语言支持 | P0 | 中文、英文界面 |
| 开机自启 | P1 | 可选的登录启动项 |

### 2.2 详细功能说明

#### 2.2.1 输入法枚举

**功能描述**：应用启动时自动扫描系统中所有已安装的输入法，并以列表形式展示。

**实现要点**：
- 使用 `TISCreateInputSourceList` API 获取输入法列表
- 提取每个输入法的名称和唯一标识符（ID）
- 支持系统内置输入法和第三方输入法
- 输入法列表按系统默认顺序排列

**用户操作**：
- 左键点击菜单栏图标展开面板
- 查看所有可用输入法列表

#### 2.2.2 一键锁定

**功能描述**：用户点击任意输入法项后，系统立即切换到该输入法并进入锁定状态。

**实现要点**：
- 使用 `TISSelectInputSource` API 执行切换
- 切换成功后设置 `isLocked = true`
- 保存锁定输入法的 ID 用于后续比较

**用户操作**：
- 点击输入法列表中的任意一项
- 应用切换到该输入法并锁定
- 菜单栏图标变为 🔒

#### 2.2.3 锁定状态管理

**功能描述**：提供全局锁定/解锁切换按钮，用户可随时暂停或恢复锁定功能。

**实现要点**：
- 解锁状态：监听但不纠正输入法变化
- 锁定状态：检测到变化立即纠正回锁定目标
- 状态变化时更新菜单栏图标

**用户操作**：
- 点击面板底部的锁定/解锁按钮
- 或通过右键菜单快速切换

#### 2.2.4 菜单栏常驻

**功能描述**：应用在 macOS 状态栏常驻，显示当前锁定状态。

**图标状态**：
- 🔓 未锁定：输入法可自由切换
- 🔒 已锁定：输入法被固定

**交互方式**：
- 左键点击：展开主面板，显示输入法列表和锁定状态
- 右键点击：显示快捷菜单（锁定/解锁、退出）

#### 2.2.5 多语言支持

**支持语言**：
- 简体中文 (zh-Hans)
- English (en)

**实现方式**：
- 使用 `.lproj` 本地化文件
- 自动跟随系统语言，或用户手动选择

#### 2.2.6 开机自启

**功能描述**：用户可选择是否在登录时自动启动应用。

**实现要点**：
- 使用 `ServiceManagement` 框架注册登录项
- 提供开关选项在设置面板

### 3.3 核心实现原理

#### 3.3.1 监听-纠正模式

由于 macOS 没有提供真正的输入法锁定 API，产品采用"监听-纠正"变通方案：

1. **监听阶段**：通过 `DistributedNotificationCenter` 监听系统输入法变化通知
2. **检测阶段**：比较当前输入法 ID 与锁定目标 ID
3. **纠正阶段**：检测到不一致时，延迟 50ms 后调用 `TISSelectInputSource` 强制切回

#### 3.3.2 关键参数设计

| 参数 | 值 | 说明 |
|------|-----|------|
| 纠正延迟 | 50ms | 确保系统完成切换后再纠正 |
| 比较标识 | InputSourceID | 唯一标识符，避免同名误判 |
| 处理队列 | .main | 主线程异步处理，保证 UI 响应 |

### 3.4 系统兼容性

| macOS 版本 | 支持状态 | 说明 |
|-----------|---------|------|
| 26.2 | ✅ 完全支持 | 目标测试版本 |
| 15.0+ (Sequoia) | ✅ 完全支持 | 最低支持版本，TIS API 可用 |
| 10.5+ | ✅ 理论支持 | TIS API 从 10.5 开始提供，未测试 |
| < 10.5 | ❌ 不支持 | TIS API 不存在 |

**API 历史说明**：
- TIS API 自 Mac OS X 10.5 (2007 年) 开始提供
- 属于 Carbon.framework 的 HIToolbox 子框架
- Carbon 框架已 deprecated，但 TIS API 在当前版本仍可正常工作
- API 源码参考：[TextInputSources.h](https://github.com/phracker/MacOSX-SDKs/blob/master/MacOSX10.6.sdk/System/Library/Frameworks/Carbon.framework/Versions/A/Frameworks/HIToolbox.framework/Versions/A/Headers/TextInputSources.h)

## 10. 风险与应对

### 10.1 技术风险

| 风险 | 影响 | 概率 | 应对 |
|------|------|------|------|
| Carbon 框架被移除导致 TIS API 失效 | 高 | 低 | 持续关注 WWDC 和系统更新通知，准备迁移方案 |
| macOS 系统行为变化影响监听机制 | 中 | 中 | 充分测试每个系统版本，及时适配 |
| 第三方输入法兼容性问题 | 中 | 高 | 广泛测试，收集用户反馈 |

**技术债务说明**：
- 当前使用的 TIS API 位于 Carbon.framework 中，该框架已被 Apple 标记为 deprecated
- 截至 macOS 26.2，TIS API 仍可正常工作，Apple 未提供替代 API
- 如果未来 TIS API 被移除，可能需要：
  1. 使用私有 API（不推荐，存在安全风险）
  2. 使用 AppleScript/JXA 等方式（效率较低）
  3. 等待 Apple 提供新的公开 API
